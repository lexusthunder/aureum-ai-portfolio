The eval harness is the product

Without frozen tickets + golden outputs, prompt engineering becomes superstition. I run 50 tickets on every PR. Here's exactly what to put in yours.

Every AI engineering team I've talked to has the same workflow:

  1. Engineer changes a prompt
  2. Engineer eyeballs five outputs
  3. "Looks better" → ship
  4. Three days later: a customer hits a regression
  5. Engineer goes back, eyeballs five different outputs, ships again

This isn't engineering. It's tarot reading with a model behind it.

What an eval harness actually is

It's a locked set of inputs + scoring function that runs on every prompt change and refuses to let regressions ship.

For Aureum Genesis, my harness has 5 starter tickets (publicly visible) and 50 in production. Each ticket has:

  • A natural-language ticket text — frozen forever
  • Expected file paths touched + new files created
  • A minimum quality score (8/10 for medium, 9/10 for hard)
  • A maximum budget cap (so a runaway agent can't cheat the test)

The runner produces a JSON report. CI exits non-zero if any ticket regresses. The PR is blocked. No exceptions.

Anatomy of one ticket

- id: T005
  name: optimize-n-plus-1
  difficulty: hard
  ticket: |
    The /api/v1/properties endpoint has an N+1 query: each property
    loads its agent in a separate query. Refactor to use a single JOIN.
    Add a regression test that asserts query count <= 2.
    Keep the response schema identical.
  expected:
    touched_files: ["app/routers/properties.py", "app/models/property.py"]
    new_files: ["tests/test_properties_perf.py"]
    min_quality_score: 9
    max_budget_usd: 1.50
    min_perf_improvement: "5x query count reduction"

Notice what's not here: the ticket doesn't tell Genesis how to fix the N+1. It states the constraint (≤ 2 queries), the deliverable (a regression test), and the perf improvement (5×). The agent has to figure out the rest. That's the test.

Why this is the product, not "tooling"

Three reasons.

1. It's the only thing that makes prompt iteration safe

I changed the routing table 14 times this month. Each change could have regressed quality. The only reason I felt safe was: python eval/run_eval.py on every PR. 47/50 → keep iterating. 39/50 → revert.

2. It's how I sell on-prem to enterprise

An enterprise buyer doesn't ask "is your prompt good?". They ask "can you prove the prompt won't regress?". The answer is the eval harness, not philosophy. I show them the suite, the CI integration, the badge. Conversation flips from "trust me" to "look at the proof".

3. It's the input to the next moat

Once you have 1000 frozen tickets that Genesis passes consistently, you have a private dataset for fine-tuning your own Coder model. That dataset becomes the moat. Without the eval harness from day one, you can't build it.

What goes wrong (and how to avoid)

Pitfall 1: The eval suite drifts

Engineer changes a ticket because "it was unfair anyway". Now the regression bar is whatever the engineer thought it should be that day. Fix: tickets are append-only. New tickets fine; existing tickets frozen. Versioned in git, signed by reviewer.

Pitfall 2: LLM-as-judge cheats itself

If the same model that wrote the diff is the one scoring it, you'll get inflated scores. Fix: use a different model family for the Critic (we use Sonnet for review on diffs written by GPT-4o). Even better: use 2 critics and average.

Pitfall 3: All tickets are easy

Easy tickets pass on a coin flip. Real signal lives in the hard ones. Fix: 30% easy / 50% medium / 20% hard. The hard ones decide whether you ship.

Pitfall 4: Everyone forgets to run it

Fix: CI gate. The eval is in the merge requirements. PR can't merge without it. No human discipline required.

The 5-ticket starter set

If you're building anything agentic, copy this set:

  1. Easy: Add a health-check endpoint with tests
  2. Medium: Add rate limiting middleware
  3. Medium: Refactor logging to structured JSON
  4. Medium: Add cursor-based pagination
  5. Hard: Detect + fix N+1, regression-test query count

You can see the full YAML in the Genesis repo. Steal it.

The bigger pattern

Every category-defining AI company has a private eval set. Anthropic has theirs. OpenAI has theirs. Cognition Labs has theirs. The pattern is universal: the eval harness is the artefact you guard most carefully, because it's the only thing that converts "the model felt better today" into "we measurably shipped a better product on Tuesday at 14:30".

If you're not running an eval gate on prompt changes, you're not yet doing AI engineering. You're doing AI gambling.
