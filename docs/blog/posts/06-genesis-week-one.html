Genesis · Week 1: from skeleton to v1.0 in 72 hours

I shipped Aureum Genesis v1.0 — an autonomous AI software engineer — in a weekend. Real LLM clients, real GitHub PR opener, real Docker deploy, full sales kit. Here's the build log.

72 hours. Skeleton → v1.0. Real Claude + GPT-4o clients. Real subprocess sandbox. Real GitHub PR opener via REST API. Docker Compose. Eval harness with frozen tickets. Sales kit with 4-tier pricing and an ROI calculator. Operations runbook.

Here's how it went, hour by hour.

Hour 0 — The hypothesis

I'd been reading about Cognition Labs raising $175M for Devin at a $2B valuation. Cursor at $2.6B. Magic.dev at $1.5B. The thing they all share: autonomous coding agents. The wedge between them is positioning — IDE vs. session-based agent vs. per-PR pricing.

I picked per-PR pricing. Customer pays $X per merged PR. Predictable. Aligned to outcome. Lets us start with one repo, one ticket type, and grow.

Hours 1-6 — Architecture

Drew the 5-agent loop on paper:

Researcher → Coder → Tester → Critic → Deployer
                          ↑          ↓
                          └──── revise if score < 8

Picked Pydantic v2 for typed agent IO. Picked async Python for orchestration. Picked pgvector for the RAG layer. Picked Anthropic + OpenAI as primary LLM vendors with Gemini optional. Picked subprocess for sandboxing (E2B/Modal as the production swap).

Constraint imposed up front: hard per-ticket budget cap. The orchestrator must halt deterministically when cost exceeds threshold. This is the single rule that prevents an agent from going feral and burning $200 in 30 minutes. Wrote a 30-line CostTracker with a BudgetExceeded exception.

Hours 7-18 — Skeleton

Wrote the orchestrator. Wrote stubs for all 5 agents with mocked LLM responses (deterministic strings). Wrote the LLM router with a 15-entry routing table mapping (agent, complexity) → model. Wrote the cost tracker with real $/1M-token prices for Anthropic + OpenAI models.

By hour 18 I had a system that ran end-to-end with mock LLM responses. Pydantic objects flowed correctly through 5 agents. Eval harness scaffolded with 5 frozen tickets.

Test count: 6 unit tests. They all passed. None of them touched a real LLM yet.

Hours 19-30 — Real LLM clients

Hooked up real Anthropic SDK. Hooked up real OpenAI SDK. Each vendor wrapped in a tiny class with a complete(system, user) contract so the router could swap them transparently.

First real run on ticket T001 ("Add /api/v1/health/db endpoint, FastAPI"). Researcher produced a clean plan. Coder produced a diff that almost worked — missed one import. Critic caught it. Coder revised. Diff applied clean. Tester wrote 3 tests, all passed. Critic scored 9/10. Deployer wrote a PR title.

Cost: $0.41. Wall time: 92 seconds. From "no code" to "tested PR-ready diff" in 92 seconds. I sat there staring at the terminal output for a while.

Hours 31-42 — Sandbox + GitHub PR

Wrote subprocess-based shell tool with timeout, cwd guard, env injection, pytest output parser. Wrote fs tool with path-escape guard (refuses writes outside repo). Wrote git tool wrapping git diff, git apply, git checkout, git push via subprocess.

Wrote real GitHub PR opener using httpx + POST /repos/{owner}/{repo}/pulls. Tested against a private demo repo. PR opened. Title clean. Body well-structured.

That was the moment Genesis stopped being a toy.

Hours 43-54 — Docker + observability

Wrote Dockerfile (Python 3.11-slim + git). Wrote docker-compose.yml with Postgres + pgvector + Genesis container, healthchecks. Tested locally — full clean boot in 18 seconds.

OpenTelemetry hooks added to every agent call. Span attributes: tokens in, tokens out, cost USD, model name, agent name. In Cloud Trace, every ticket becomes a hierarchical waterfall.

Hours 55-66 — The sales kit

This is the part most engineers skip. I refused. If you don't write the sales kit, you don't have a product — you have a demo.

Wrote:

  • 30-second pitch
  • 4 pricing tiers ($49/PR Starter → $1M/yr Enterprise)
  • ROI calculator with concrete math (4×-12.6× ROI for any team > 5 engineers)
  • Cold sales email template
  • 20-minute live demo script
  • 3 deployment topologies (SaaS / VPC / on-prem air-gapped)
  • SLA template for the Scale tier
  • FAQ for the typical buyer (VP of Eng at Series B)

That document is now ~6,000 words. It's the artefact I send to design-partner candidates.

Hours 67-72 — The runbook

Operations runbook for the customer's SREs and ours. Quickstart in 5 commands. Day-2 ops (cost monitoring, killing runaways, key rotation). 5 SRE invariants (NEVER merge, NEVER read secrets, NEVER exceed budget, audit trail, rollback). On-call playbook for SEV-1/2/3 with SLA 99.5%.

Hour 72: git commit -m "Genesis v1.0 — production wiring + sales kit". Pushed.

What I learned

  1. Constraints invented up front save 10 hours later. The hard budget cap was a 30-line addition on hour 4. Without it, debugging runaway loops would have eaten my Sunday.
  2. Mock first, real second. Spent the first 18 hours with mocked LLMs. Wired real LLMs only after the orchestrator was already correct. Saved an unknowable amount of API spend on bug iteration.
  3. The sales kit is a forcing function for clarity. If you can't write the cold email, you don't know who you're selling to.
  4. 72 hours is enough. Most "MVP in a weekend" stories are bullshit because they skip the boring parts (Docker, observability, runbook). Genesis isn't bullshit because the boring parts are in the same commit as the demo.
