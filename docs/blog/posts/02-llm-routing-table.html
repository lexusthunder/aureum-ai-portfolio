How I cut Claude API costs 4.4× with a 15-line routing table

One claude-opus-4-6 everywhere is fine for a demo, lethal at scale. The fix: a routing table that maps (agent, complexity) → model. Here's the exact one I ship in Aureum Genesis.

The first version of Aureum Genesis had one configuration: every agent called claude-opus-4-6. Why? Because it was the smartest model and I wanted the system to work before I optimised. Standard play.

Cost per ticket on Opus-everywhere: $0.80.

That's fine when I'm running 5 tickets a day to demo to friends. It's lethal when a customer pushes 200 tickets through Genesis on day one and gets a $160 bill before lunch.

The insight

Different agents have radically different cognitive loads.

  • The Researcher reads a ticket and produces a typed plan. It needs reasoning, not raw code-gen power. Sonnet 4.6 is enough.
  • The Coder writes diffs. It needs strong code generation. GPT-4o is great here. Opus only for hard tickets.
  • The Tester generates pytest cases. This is mostly templated string work. Haiku handles it for a tenth of the price.
  • The Critic reviews diffs on a 1-10 scale. Needs judgement, not creativity. Sonnet.
  • The Deployer writes a PR title + body from a structured input. Haiku again.

The routing table

15 lines, no magic:

ROUTING_TABLE = {
    ("researcher", "low"):    "claude-haiku-4-5",
    ("researcher", "medium"): "claude-sonnet-4-6",
    ("researcher", "high"):   "claude-opus-4-6",
    ("coder",      "low"):    "gpt-4o-mini",
    ("coder",      "medium"): "gpt-4o",
    ("coder",      "high"):   "claude-opus-4-6",
    ("tester",     "low"):    "claude-haiku-4-5",
    ("tester",     "medium"): "claude-haiku-4-5",
    ("tester",     "high"):   "claude-sonnet-4-6",
    ("critic",     "low"):    "claude-sonnet-4-6",
    ("critic",     "medium"): "claude-sonnet-4-6",
    ("critic",     "high"):   "claude-opus-4-6",
    ("deployer",   "low"):    "claude-haiku-4-5",
    ("deployer",   "medium"): "claude-haiku-4-5",
    ("deployer",   "high"):   "claude-haiku-4-5",
}

Complexity is set by the Researcher's output (a string field estimated_complexity on the typed plan). Coder runs at the complexity the Researcher set; everyone else runs at medium by default with overrides for hard tickets.

The result

Cost per ticket dropped from $0.80 to $0.18. Same quality on the eval harness — 47/50 tickets passing on both configurations, identical Critic scores.

That's 4.4× cheaper. For a 100-customer SaaS at 200 tickets/customer/month, the math is real money:

Before: 100 × 200 × $0.80 = $16,000 / mo  →  $192k / yr LLM spend
After:  100 × 200 × $0.18 = $ 3,600 / mo  →  $ 43k / yr LLM spend
Saved: ~$148k / yr  =  one engineer's salary

What I learned

  1. Optimise for the median ticket, not the worst. 80% of tickets are easy/medium. Spending Opus money on every one is wasteful.
  2. The eval harness gates the routing change. I would not have shipped this without 50 frozen tickets confirming "same quality, lower cost". Routing changes are prompt changes — they need eval gates.
  3. Different vendors for different jobs. Anthropic's Sonnet beats GPT-4o on judgement tasks (Researcher, Critic). GPT-4o beats Sonnet on raw code-gen (Coder). Mixing vendors isn't disloyal — it's correct.
  4. Pricing changes monthly. The routing table lives next to the price table. When Haiku drops 30% next quarter, I move borderline tasks to Haiku.

What I would extend

The current router is a static table. The next version is a learned router: a small classifier (could be sklearn) that takes ticket features (length, complexity flags, file count touched) and predicts which (agent, model) combination clears the eval bar at lowest cost. Train it on every ticket Genesis processes. Re-train weekly.

The bet: a learned router compresses cost another ~2× without quality loss. That's the difference between a 30% gross margin and a 60% gross margin SaaS.
